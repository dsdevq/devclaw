# Changelog

## Unreleased

### Changed
- **Execution-related modules grouped under `devclaw/engine/`.** Seven top-level modules — `engine.py`, `sandcastle_runner.py`, `claude_sdk_engine.py`, `host_runner.py`, `stub_engine.py`, `runner_io.py`, `workspace.py` — moved into one package, renamed without their `_runner` / `_engine` suffixes: `engine/__init__.py` (the protocol), `engine/sandcastle.py`, `engine/claude_sdk.py`, `engine/host.py`, `engine/stub.py`, `engine/runner_io.py`, `engine/workspace.py`. Reduces top-level chatter from 22 modules to 16 and puts everything that *executes the work* under one folder, so a reader looking for "how does devclaw run a task" finds it in one place instead of seven. Public imports preserved (`from devclaw.engine import EngineRequest` still works — it's a package now, and the protocol re-exports from `__init__.py`); internal call sites updated. The e2e trace harness, run before and after the move, produces byte-for-byte identical cognition-prompt hashes — proving the runtime path is unchanged.

### Added
- **Prompts moved out of Python — `devclaw/prompts/*.md` is the single source of prompt truth.** Prompt strings used to live as Python triple-quoted constants scattered across nine modules. Editing prose required a Python diff, a code review, and a CI cycle, and a prompt change was visually buried inside unrelated Python. Now every system prompt + JSON contract is a `.md` file under `devclaw/prompts/`: `plan-goal`, `plan-spec`, `goal-planner`, `goal-evaluator`, `scope-grill` (+ `scope-grill-contract`), `eval-judge`, `review-gate`, `research-discovery`, `owner-summary`, and the four sandcastle-engine task templates (`sdk-implement-feature` / `sdk-fix-bug` / `sdk-review-repository` / `sdk-onboard`). A tiny loader (`load_prompt(slug, **vars)`) reads the file (cached for the process lifetime), runs `str.format` so JSON literals stay legible via `{{ }}` escaping, and returns the prompt verbatim. Composite prompts (the grill assembles rules + transcript + contract) keep their Python builder — the static blocks live in `.md`, the dynamic plumbing stays in code. The runtime path is preserved exactly: the e2e trace harness, run before and after this refactor, produces a byte-for-byte identical timeline — every cognition prompt hashes the same. Two fixes the harness surfaced and the new regression test pins: a previous output dir reused across runs accumulated `log.md` and shifted the planner-prompt hash (the harness now cleans `stub-goals/` per invocation), and a `tests/test_e2e_trace.py::test_two_runs_produce_identical_prompt_hashes` now guards against the same class of hidden-state regression.
- **Cognition seam — one chokepoint behind which the LLM hides.** Every cognition call previously hardcoded `claude --print` at the call site through `claude_with_model`. Swapping in another backend (different model family, local model, HTTP API, recorded fixture for evals) would have meant touching every role module. New `devclaw/cognition.py` introduces a `Cognition` protocol with two implementations: `ClaudeCognition` (the current subprocess, default) and `StubCognition` (canned responses keyed by role, for offline harnesses + evals). Selection is by env: `DEVCLAW_COGNITION=claude` (default) or `stub`; unknown values fail loud at first use. `claude_with_model(model, *, role)` is now a thin shim that delegates to `cognition.bind(model, role=role)` — historical name and signature preserved, so no role module changed. Future backends (HTTP API, local model, fixture-replay) drop into this seam with ~50 lines and no caller edits. Covered by `tests/test_cognition.py` (env selection, role-keyed stub responses, runtime swap, trace integration); the existing e2e trace harness produces a byte-for-byte identical timeline before and after the swap, proving the runtime path is preserved.
- **E2E trace harness — see what the chef actually did, step by step.** The refactor-blindness problem: today a change can ship green unit tests but quietly break the *runtime* path (wrong prompt, missing tick, an extra cognition call) because the suite uses fakes and never watches the whole flow. New `devclaw/loom/trace.py` adds a `Tracer` that observes the goal lifecycle fire-and-forget over a contextvar — every cognition call (role, model, prompt hash + preview, response preview, latency, error), every tick (incoming lifecycle/phase → outgoing outcome), every dispatch (tool, ref id, discovery/done-check flags), every delivery (gate verdict, PR url), every owner-altitude notification. `record_*` is a no-op when no tracer is attached, so production stays untouched; tests/harness attach one to record. Each cognition role labels its `claude_with_model(..., role=…)` factory call (planner / evaluator / grill / judge / summary / review / goal_planner) so the trace is groupable. `evals/e2e_trace.py` drives a goal end-to-end in **stub mode** (FakeClaude + FakeEngine, deterministic, no network — runs in CI) or **live mode** (real `claude --print`, real engine — ticks an existing goal in `DEVCLAW_GOALS_DIR`) and emits both `trace.json` (machine-readable, diffable across runs) and `timeline.md` (human-readable per-tick narrative). The harness is the safety net subsequent refactors rely on: if a refactor changes the runtime shape, the trace shape changes, and that's visible. Covered by `tests/test_e2e_trace.py` (full lifecycle from investigating → done; round-trip of the JSON + timeline format).

### Changed
- **Grill split: the chef owns the craft (which questions matter), the OpenClaw waiter owns the conversation.** The old in-chef `grilling` lifecycle phase — a durable per-goal Q&A loop reusing the spec-kit elicitation under `DEVCLAW_GOAL_GRILL=1` — was dead code post-elicitation-removal: the Telegram answer channel that fed it was never wired and the cognition tangled "knowing what to ask" with "holding the conversation." Two skills mixed together. Now split cleanly: the **chef** exposes one stateless MCP tool, **`scope_grill(idea, transcript) → {ask|done}`**, that runs one turn of the elicitation cognition (kept verbatim — the methodology adapted from Matt Pocock's MIT `grill-me` still applies); the **waiter** (the OpenClaw devclaw agent on the VPS) holds the conversation in the Telegram chat, calls `scope_grill` each turn, and once it returns `{"action":"done","spec":…}` files the order with `create_goal(spec=…)`. `create_goal` gains an optional `spec:str` param that's persisted to `<goal>/spec.md` so the evaluator judges done against the shared contract. Deleted: `devclaw/goal/grill.py`, `goal/tick.py::_run_grill` + `Phase.GRILLING`, `goal/store.py::{read,write}_grill` + `answer_pending`, the `grilling` lifecycle, the GRILLING branch in `_resolve_discovery` (investigation now flows straight into executing), `DEVCLAW_GOAL_GRILL` + `DEVCLAW_GOAL_GRILL_MODEL`. Kept: `elicitation.py` (repackaged as MCP-callable, lazy tier binding via `default_caller`), `DEVCLAW_GRILL_MODEL`, `DEVCLAW_MAX_GRILL_QUESTIONS`, `read_spec`/`write_spec`. The chef no longer has an asynchronous in-process question loop — the painful part — and the line between *craft* and *conversation* is now legible. Covered by `tests/test_elicitation.py` + `tests/test_create_goal_warnings.py` (spec roundtrip).

### Fixed
- **A durable goal re-shipped + re-announced a finished milestone forever when the next-action planner hit a usage limit (the "duplicate merged-PR notification" loop).** Found dogfooding CloseLoop: after M2 merged, the goal couldn't plan M3 (session cap), and every 15-min tick it re-consumed the *already-done* M2 task — re-running the no-op merge, re-sending a "shipped + merged" ping that pasted the **entire M2 instruction prompt** — then crashed before advancing. Four linked defects, all fixed: (1) **persist bug** — `goal_tick` computed `in_flight=None` in memory but didn't `save_status` before the planner call, so a planner crash aborted the tick with the stale pointer on disk and the next tick re-processed it; now the consumed action is persisted *immediately*, before any downstream cognition. (2) **wrong exception caught** — the goal planner shells out via the shared `claude --print` caller, which raises `planner.PlannerError` (not `GoalPlannerError`) on a non-zero exit; `goal_tick` caught only `GoalPlannerError`, so a usage limit *escaped* to the outer "tick error (isolated)" handler instead of pausing — now both are caught. (3) **unclassified wording** — `"You've hit your session limit · resets 12:20am"` (the Claude Code 5-hour session cap) wasn't in the quota patterns, so it would never have paused; added. (4) **notification pasted the full prompt** — the ship ping used the action's raw `goal` (a long instruction) because the plain-language summarizer fails open to raw text under quota; it now uses a terse first-line label by construction. Covered by `tests/test_goal_tick.py` (persist-before-crash, `PlannerError` caught-not-escaped, concise notification) + `tests/test_limits.py` (session-cap wording).
- **The post-gate diff guards (test-integrity + the new review gate) were silently no-op in the deployed container — they were handed a path that doesn't exist in the server's mount namespace.** After the verify gate passes, `_run_and_settle` computes the change's `git diff` to feed the test-integrity guard (and now the review gate). That git runs **in the server process** (the `devclaw-mcp` container), but the path was passed through `_translate_workspace_path` — the helper that maps a workspace's *container* path (`/var/lib/devclaw/workspaces/…`) to its *host* path (`/srv/devclaw/workspaces/…`) for the sandbox's `docker -v` bind mount. The host path doesn't exist inside the server container, so `git -C /srv/…` failed and `_git_diff` returned `""` — an empty diff means the test-integrity guard sees no test changes (always "clean") and the review gate sees nothing to review (skips). So a change could delete/weaken tests and still settle `done` on a live deployment. **The suite never caught it** because test workspaces are `tmp_path`s that don't start with the container prefix, so the translation was a harmless no-op locally — it only bit in production. Surfaced by dogfooding the review gate end-to-end (the review never fired; the diff was empty). Fix: the in-process git diff uses `workspace_dir` **verbatim** — translation is only for the docker bind mount. Regression test pins the path the diff is taken against (`tests/test_review_gate.py`).
- **Root cause of the long-held write lock: `ProjectRegistry.create()` leaked an open transaction on a duplicate id.** A duplicate `create()` raised `ProjectExists` on the `INSERT`'s `IntegrityError` *without rolling back* — and pysqlite (deferred-transaction mode) does not auto-rollback, so the failed statement's transaction stayed open on the long-lived server connection, holding the SQLite write lock until the connection's next commit. That starved every other connection's writes (this is what kept the CLI delete locked for 75 s during the dogfood — it cleared only when a later in-server write committed). `create()` now rolls back on `IntegrityError` before raising. Regression test reproduces the exact stall (a second connection's write blocks until busy_timeout) and confirms the fix releases it. This is the structural fix; the `busy_timeout` change above is the safety net.
- **The control-plane CLI's writes no longer fail with `database is locked` while the server is running.** The MCP server and the `devclaw` CLI each open their own SQLite connection to the same db file, but neither `state_store` nor `project_registry` set a `busy_timeout` — so with SQLite's default of `0`, a CLI write (`projects register/update/link/rm`) issued while the server held the write lock failed *instantly* with `sqlite3.OperationalError: database is locked` instead of waiting its turn. Found dogfooding: deleting a throwaway registry row from the CLI failed continuously for 75 s+ while the goal layer was active, then succeeded the moment the server's write committed; reads were never affected, which made the CLI *look* healthy. On a live deployment (server always up) this made the CLI's write subcommands effectively unusable. Both connections now set `PRAGMA busy_timeout` (default 5000 ms, `DEVCLAW_SQLITE_BUSY_TIMEOUT_MS`), so a contended writer waits for the lock and succeeds. Covered by `tests/test_project_registry.py` (a pragma-value check + a two-connection test that holds the lock in one thread and asserts the other's write waits then wins) and `tests/test_state_program.py`.

### Changed
- **Delivered PRs now read like real engineering — the engineer writes its own commit, branches are feature-named, and the project carries an accumulated `AGENTS.md` harness.** Dogfooding CloseLoop surfaced that PR titles/branches/bodies were all derived from the task's *raw instruction text* (the goal-planner writes a long spec as the "goal"), so a PR title was literally `feat: Read PRD.md, ARCHITECTURE.md, … and all…` and the body was the whole instruction. Now: (1) the in-sandbox runner asks the engineer to **commit its own change with a clean conventional-commit message** (subject = what changed, body = why + how verified; no push/PR — devclaw delivers); (2) `delivery.py` derives the PR **title from that commit subject** and the **branch from it** (`feat/add-deal-crud`, not `devclaw/<uuid>-<goal-slug>`), with the goal-derived path kept only as a fallback for when the agent leaves the tree uncommitted (a `devclaw/*` branch, so an auto-committed change stays visibly distinct); (3) the PR body leads with **## Changes** (what changed) and collapses the long instruction into a **Ticket** `<details>`, keeping the gate verdict + diffstat + the honest review caveat. (4) The runner briefing makes **`AGENTS.md` the project's accumulated agent-harness** — read first AND kept current every task (stack, run/test, layout, conventions, decisions, gotchas, reusable patterns) so a future task doesn't re-derive the same context (directly attacks the token-burn cost). Covered by `tests/test_delivery.py` (commit-derived branch/title/body) + `tests/test_runner_wrappers.py` (self-authored commit + AGENTS.md harness instructions).
- **`loom` — the reusable orchestration core extracted under a neutral name (drop the `-claw` prefix).** First step of the planned modular extraction: a new `devclaw/loom/` package is the **extraction seam** for the engine-agnostic substrate devclaw is built on (failure classification, the test-integrity guard, the goal domain types + durable store), so it can eventually become a standalone library that any product — not just devclaw — orchestrates with. Reversible + non-breaking: the two pure self-contained utilities (`limits`, `test_integrity`) physically move into `loom/` with **thin back-compat shims** left at the old `devclaw.*` paths, while the goal domain/store are re-exported into `loom`'s curated public surface (`from devclaw.loom import classify_failure, scan_diff, Goal, GoalStore, …`) pending a physical move once a second consumer proves the seam. Internal callers now import from `loom`. Covered by `tests/test_loom.py` (public surface + shim identity).
- **Dashboard presentation split out of `server.py` into `devclaw/dashboard.py`.** The inline HTML/CSS page builders (programs · program detail · goals · goal detail · projects) moved into pure renderers that take plain data + the auth query-string and return HTML; the HTTP routes in `server.py` are now thin (fetch data → call a renderer). Removes ~250 lines of markup from `server.py` and gives the dashboard its first unit coverage — `tests/test_dashboard.py` feeds each renderer fake data and asserts the markup (pills, nav, escaping, empty states). No behavioural change.
- **The task brief now carries a code-quality bar — not just "make the tests pass."** The methodology wrapper for `implement_feature`/`fix_bug` (`openhands-runner/runner.py`) previously briefed the agent to get its bearings + verify, but said nothing about *quality* — so the agent optimized for the one thing it was told to satisfy (a green suite) and shipped "a working version": logic inlined wherever instead of in the right module, happy-path-only tests, even dead/no-op code that passes because nothing exercises it (a real pass-rate run produced a `Directory.Enumerate(...).Take(0).Count()` "accessibility check" that enumerates nothing and never throws — green, but meaningless). The wrapper now adds an explicit bar: write code you'd approve in review, match the surrounding architecture (put logic where similar logic lives), follow existing style/error-handling, **no dead/no-op code**, handle real edge/error cases, make added tests genuinely exercise behaviour (incl. failure cases) without weakening existing ones, and — since *a green gate is necessary but not sufficient* — re-read your own diff critically before finishing. This is the quality expectation devclaw (the PM) owes the engineer; repo-specific conventions still come from the repo's own `AGENTS.md`. Covered by `tests/test_runner_wrappers.py`.
- **Delivered PRs now describe themselves.** `delivery.py` used to set the PR/commit **title to the raw goal string truncated mid-word** (`…the single cron with t`) and the body to one line of boilerplate ("Delivered by devclaw. Verified by the task's gate.") — useless for the reviewer who's supposed to *review* it. Now the title is a clean conventional-commit summary (`feat:`/`fix:` from the task kind, backticks stripped, truncated on a word boundary), the branch slug no longer cuts mid-word, and the body carries **what** (the ticket), **verification** (the gate command + verdict), **files changed** (the diffstat), and an honest caveat that a green gate means the tests pass, not that the code is right. Threaded `kind` + the gate verdict from the task into `deliver_change`. Covered by `tests/test_delivery.py`.

### Added
- **Pre-PR adversarial diff-review gate — make "green" mean "trustworthy".** The verify gate proves a change *behaves* (tests pass) and the test-integrity guard proves it didn't go green by gutting the suite — but neither one *reads the code*. That's the hole a spectator-PO can't cover: a `.Take(0)` dead-code line, a happy-path-only implementation, logic in the wrong layer, or a frontend change the backend gate never exercises all sail through a green gate. New `devclaw/review_gate.py` closes it — after the verify gate **and** test-integrity pass but **before** the PR opens, a separate `claude` pass (host-side OAuth, tiered via `DEVCLAW_REVIEW_MODEL`, default sonnet) reads the diff against the ticket + the same `_QUALITY_BAR` the engineer was briefed with, and returns a structured verdict (`approve` / `request_changes` with located issues). On `request_changes` the issues are fed back through the **existing** retry loop (`task_queue._run_and_settle` / `DEVCLAW_MAX_RETRIES`) exactly like a gate failure — the agent gets a bounded chance to fix the named defects, then it escalates to the owner. It sits next to the existing gates (the diff is computed once and shared with the test-integrity guard), applies only to code-producing kinds (`implement_feature`/`fix_bug`), and **fails open** — a disabled gate, an empty diff, or any reviewer error never blocks a change that already passed the behavioural gate. `DEVCLAW_REVIEW_GATE=0` disables it (escape hatch + quota lever; it costs one `claude` call per successful code task). The verdict/issue list is self-reconciling: `request_changes` ⇔ at least one blocker/major issue, so a style nit can't trap a task in the retry loop and an "approve" that names a blocker is upgraded. Covered by `tests/test_review_gate.py` (16 tests: prompt build, verdict reconciliation, feedback formatting, JSON parsing, and the queue integration — request_changes retries with the feedback then ships, persistent request_changes escalates, approve ships first try, the gate fails open on a reviewer error, and is skipped when disabled / for non-code kinds).
- **Project registry + control plane — one source of truth for "what is devclaw working on", drivable from chat / API / CLI.** DevClaw already had three lower-level primitives — ephemeral tasks + programs (SQLite) and durable goals (on disk) — but no first-class entity tying a *repo* to the goal(s) driving it, its workspace, and its live preview. New `devclaw/project_registry.py` adds a thin **`Project`** record (id slug, name, repo_url, workspace_dir, preview_url, status active|paused|archived, linked goal_ids, notes) in its own `projects` table on the shared SQLite file. It is deliberately **thin and rot-proof**: it links goals *by id* and joins their live status on read (`project_rollup` → each goal's phase/direction + a derived `health`: working/blocked/done/idle/archived), never caching goal phase. Three faces of the same control plane: **MCP tools** `register_project` / `list_projects` / `project_status` / `update_project` / `link_goal` / `delete_project` (distinct from the build-from-scratch `build_project`/`get_project` interview tools — two different "project" concepts, kept namespaced apart); a **CLI** (`devclaw projects list|show|register|update|link|archive|rm`, also `python -m devclaw.cli`) that talks to the same stores directly so it works without the server running; and a **`/projects` dashboard** portfolio view. Covered by `tests/test_project_registry.py` + `tests/test_cli.py` (23 tests).
- **Goal-level no-progress watchdog — catches a goal that keeps churning without shipping.** The per-task timeout (`DEVCLAW_TASK_TIMEOUT_S`) kills one hung *run*, but a goal can stay busy — dispatching, failing the gate, re-planning, dispatching again — for a long time while never delivering, and nothing surfaced that. The heartbeat now runs a zero-token wall-clock check first on every tick (including while an action is still in-flight — pure timestamp math, no cognition): an **executing** goal that goes `DEVCLAW_GOAL_NO_PROGRESS_S` (default 21600s = 6h, `0` disables) without a delivery gets **one** OWNER-altitude ping, then stays quiet until the next delivery resets it (once per stall, not every tick). It self-initializes its baseline the first time it sees an executing goal (covers legacy goals), and stays silent for goals already waiting on the owner (blocked / grilling / plan_review). Two new `GoalStatus` fields (`last_progress_at`, `no_progress_notified`, persisted in `STATUS.md`), surfaced via `get_goal`/`tail_goal` under `progress`. Covered by `tests/test_no_progress.py`.
- **The goal layer — DevClaw is now the software-development PM: it owns durable goals, drives them to PRs, and EVALUATES direction (the folded-in goalclaw).** The former standalone `goalclaw` service is retired and its logic moved in-process. A `program` is still a bounded one-shot DAG; a new **`goal`** is an open-ended standing intent advanced across a heartbeat, steerable and direction-evaluated, persisted on disk under `DEVCLAW_GOALS_DIR` (`<id>/goal.yaml · STATUS.md · log.md · inbox.md · deliveries.md`) — ephemeral body, durable mind. Five new MCP tools: `create_goal` / `get_goal` / `list_goals` / `steer_goal` / `evaluate_goal` — register a goal, ask what's going on / what direction, correct it, force an evaluation. **The fold deletes a whole transport + race class:** goalclaw dispatched over HTTP MCP and re-derived "delivered" from a camelCase blob + a `/wake` callback; in-process, dispatch is a function call (`goal_engine.InProcessEngine` → the same `TaskQueue`), poll is a SQLite read, and a settling task fires `TaskQueue.set_on_settle` to wake the goal heartbeat immediately — so the bearer token, the `/wake` endpoint, and the "polled `done` before `pr_url` was written" race are all gone by construction. **Real direction evaluation** (the hard part, now that the evaluator sits next to the repo + execution context): each finished action's *full* result (agent output + gate output) is captured to `deliveries.md` at zero token cost; a periodic `goal_evaluator` LLM call judges whether the *delivered work* is achieving the objective (not by counting backlog items) and writes corrections back as steering; and the planner's `done` is only a **proposal** — it triggers a read-only `review_repository` against `done_when` and the goal closes **only if the evaluator confirms `achieved`** from that grounded review. "Done" is gated on evaluation, not on PR-counts. The zero-token idle guard carries over intact (idle + in-flight-running ticks cost 0 `claude` calls). Cognition reuses devclaw's own `claude --print` callers at new `DEVCLAW_GOAL_PLANNER_MODEL` / `DEVCLAW_GOAL_EVAL_MODEL` tiers (sonnet) — one cognition mechanism, no Agent-SDK dep. New modules `goal_{service,tick,planner,evaluator,engine,store,models}.py` + `workspace.py`; new env `DEVCLAW_GOAL_{TICK_SECONDS,EVAL_EVERY,VERIFY_DONE,NOTIFY_URL}`; `pyyaml` added. Covered by `tests/test_goal_{store,planner,evaluator,engine,tick}.py` (45 tests; the quota guard pins planner+evaluator at 0 calls on idle).
- **Measured pass-rate harness + the real number: 5/5 = 1.0 on a real repo.** `evals/measure_passrate.py` runs a basket of real backend tickets on lifekit-dashboard through the **real** engine (docker sandbox + `devclaw-sandbox-dotnet` + `claude`, wired exactly like the server: `TaskQueue(runner=run_sandcastle)`), gates each with `cd backend && dotnet test`, and delivers each as a PR (`open_pr`) — each task on its own fresh clone, sequential for quota safety. First run (2026-06-01, Sonnet exec): **5/5 done + gate-verified** — 4 net-new features (`GET /api/crons/{id}`, `?status=` filter on `/api/proposals`, `GET /api/gaps/summary`, `GET /api/health/ready`) and 1 hardening fix (`/api/proposals/{slug}/reject` → 404 on unknown slug), delivered as PRs #10–#14. **+19 net-new tests, 0 existing tests deleted/skipped/weakened, 0 regressions** (every gate ran the whole suite). This run also **live-validates the PR push** — the commit→push→`gh pr create` path that PR delivery previously only unit-tested. Honest scope: small-to-medium machine-verifiable backend tasks on one repo; UI / ambiguous specs still need a human. (Reports land in the git-ignored `evals/runs/`.)
- **Onboarding — `onboard(workspace_dir)` generates a draft `AGENTS.md` so future tasks start informed.** A 4th task kind that mirrors the `review_repository` read-only path but is permitted to write a single file. OpenHands inspects the repo read-only and writes its comprehension — **"what is" only**: stack, layout, how to build/run/**test** (and which command to use as the verify gate), conventions, and setup gotchas — to `AGENTS.md`. Project direction / roadmap / decision-log are deliberately out of scope (kept separate per the operating model). This is the file 1b's project-guide briefing already looks for; `onboard` generates it for repos that lack one (lifekit-dashboard has none) so the next `implement_feature`/`fix_bug` doesn't start blind. **Human-in-the-loop:** the draft is not authoritative until you review it — it lands in the repo working tree (review via `git diff`) and the agent's summary appears in `result_json`; no verify gate and no PR (it's comprehension, not a code change). If an `AGENTS.md` **already exists**, the agent validates it against the real repo and **keeps what's still accurate**, correcting only what's wrong/stale/missing — it does not clobber hand-written project memory. New `onboard` kind threaded through the existing queue/engine/runner seam (no new execution machinery); covered by `tests/test_onboard.py`.
- **PR delivery — a verified change comes back as something you review, not produce.** `implement_feature`/`fix_bug` take `open_pr=True`: when the task settles `done`, DevClaw commits the agent's change to a `devclaw/<task>-<slug>` branch, pushes, and opens a PR (via `gh`), recording the URL on the task and in the `notify_url` payload. Best-effort and **non-fatal** — a delivery hiccup never un-does a `done` task; it degrades gracefully (not a repo → skip; no changes → skip; no remote → leave a local branch; push/PR auth missing → stop at the pushed/committed state) and records what it managed. Push/PR auth is a GitHub token (`GITHUB_TOKEN`/`GH_TOKEN`) or `gh`'s own login — separate from the Claude OAuth pillar (repo access, not cognition billing). Closes the gap behind the README's "turns a goal into a verified PR" (which was aspirational). New `devclaw/delivery.py`; nullable `tasks.deliver`/`tasks.pr_url` columns; covered by `tests/test_delivery.py`. (The push+GitHub-PR path is exercised live; the local commit/branch/degradation logic is unit-tested.)
- **Retry-on-fail — the third leg of the reliability triad (verify + retry + human).** A task that fails its verify gate, or hits a transient agent error, is re-run up to `DEVCLAW_MAX_RETRIES` (default 1) — each retry feeds the failure back into the goal ("your previous attempt did not pass verification: …; diagnose and fix") so the agent self-corrects rather than repeating the same mistake — then escalates (the `notify_url` fires on the terminal `failed`). **Timeouts are not retried** (a stuck run would likely just hang again — escalate immediately). Builds directly on the verify gate: the gate catches a bad result, retry gives a bounded second chance. `CancelledError` still propagates, so an explicit `cancel_task` is never retried. Covered by `tests/test_task_retry.py`.
- **`.NET`-capable sandbox image variant.** `.sandcastle/Dockerfile.dotnet` builds `devclaw-sandbox-dotnet` = the base sandbox + the .NET 10 SDK (python/node/claude/runner reused), selected per-run via `DEVCLAW_SANDBOX_IMAGE`. Keeps the *default* sandbox lean — .NET ships only in this variant — so the agent can build/run/test .NET repos (lifekit-dashboard, finance-sentry) and the verify gate can run `dotnet test`. (Longer-term the per-task box should derive from the project's own declared base image, keeping devclaw language-agnostic; this variant is the pragmatic first step.)

### Added
- **Per-task wall-clock timeout — a hung run fails cleanly instead of burning quota forever.** A task that exceeds `DEVCLAW_TASK_TIMEOUT_S` (default 1800s, `<=0` disables) is cancelled — which tears down its sandbox via the runner's existing teardown path — and marked `failed` with a clear reason, then notified like any other terminal state. Motivated by the live smoke, where a silent post-init agent hang ran 6+ minutes and leaked its container. Shares the cancellation/teardown path with explicit `cancel_task`, so semantics are consistent. It's a coarse backstop (catches both silent hangs and busy-loops); a finer no-progress timer that kills a *silent* hang faster is a natural follow-up. Covered by `tests/test_task_timeout.py`.

### Fixed
- **Runner stdout reader no longer crashes on a large event line.** Both engines created the runner subprocess with asyncio's **default 64 KiB** per-line StreamReader limit. The runner emits one JSON object per line (`event:` / `result:`), and a single event can exceed 64 KiB — a file read/write observation, a big diff. A real feature run hit this: the read threw "Separator is not found, and chunk exceed the limit" and the task was marked `failed` even though the agent's work was complete and correct. The engines now pass a generous `STREAM_LINE_LIMIT` (64 MiB — far above any real event, still bounded) to `create_subprocess_exec`. Found by a live `implement_feature` run on a real .NET repo (the agent's multi-file feature was verified correct + regression-free out-of-band; only devclaw's plumbing had choked). Covered by `tests/test_runner_io.py`.
- **Sandbox curated mount also needs `.claude.json` (account identity), not just the credential.** The first cut of the curated `~/.claude` allowlist defaulted to `.credentials.json` only — verified to *authenticate* (`claude --print`), but a live smoke caught the agentic ACP loop **hanging after init with no actions**: the agent had the OAuth token but not the account identity (`oauthAccount`/`userID` in `.claude.json`) the ACP server needs to act (auth ≠ agency). Default allowlist is now the identity *pair* `.credentials.json,.claude.json`. The genuine leak stays cut — `.claude.json` here carries identity + caches, no `mcpServers`, `projects` empty (skills/plugins/global-CLAUDE.md/history are still excluded). End-to-end smoke now green: agent acts, the verify gate runs in-sandbox and passes, task settles `done`.

### Added
- **Verify gate — a task is "done" only when the project's own checks pass, not when the agent says so.** Until now a task settled `done` purely on the agent loop finishing (`result: ok` meant "didn't crash," not "the goal was met"). `implement_feature` and `fix_bug` now take an optional `verify_cmd` (e.g. `"dotnet test"`, `"npm run build && npm run test:ci"`). After the agent finishes, the runner runs that command **in the workspace** — inside the sandbox for the docker engine (same toolchain the agent built with), on the host for the host engine — and attaches a verdict (`{ran, cmd, passed, exit_code, timed_out, output}`) to the result; the gate has a wall-clock cap (`DEVCLAW_VERIFY_TIMEOUT_S`, default 900s) so a hung suite can't wedge the task. `TaskQueue` then settles `done` only if the gate passed (or no gate ran); a failed or timed-out gate marks the task **failed** with the command output captured in the error, so a human — or the upcoming retry — sees what broke. The engine reports the verdict; the orchestration decides done-vs-failed (mechanism/cognition split). Threaded through `EngineRequest.verify_cmd` + a new nullable `tasks.verify_cmd` column (forward-compat ALTER). Covered by `tests/test_verify_gate.py`. (This is the mechanical gate; the wrapper's "run the tests" is the agent self-checking — defense in depth, not a substitute.)
- **Task goals are now briefed + self-verified, not passed raw to the agent.** `implement_feature` used to forward the bare goal straight to the agent, so on an existing repo it started blind — no read of the project's conventions, no check of its own work. Both `implement_feature` and `fix_bug` now prepend a methodology wrapper (`openhands-runner/runner.py`): get your bearings first (read `AGENTS.md`/`CLAUDE.md`/`README` + the surrounding code so the change matches the project), keep the change focused, then **verify** — run the project's test/build command and iterate until it passes, ending with a summary. Shaped by OpenHands' own prompting guidance (concrete, location-aware, scoped, run the tests). This is the engineer self-checking; devclaw's own verify gate (separate, upcoming) still double-checks the result. Pairs with the sandbox-config curation below: that cut the *host-global* skill/MCP leak, this adds back the *repo-scoped* briefing. Covered by `tests/test_runner_wrappers.py`. (Deliberately did **not** wire `AgentContext(load_project_skills=True)`: it defaults off, mainly surfaces `.openhands/skills/` that the target repos don't ship, and overlaps Claude Code's own cwd reading — kept in reserve.)
- **Sandbox config isolation — a curated `~/.claude`, not the host's whole personal setup.** The per-task sandbox no longer bind-mounts the entire host `~/.claude` into the engineer. It mounted that dir read-only for OAuth, but the dir holds far more than auth — personal `skills/`, `plugins/` (+ their MCP servers, which need network/auth absent in the sandbox), the global `CLAUDE.md` (which points at the unmounted `~/memory`, so its instructions are dead inside the box), and `projects/` + `history.jsonl` session state. Projecting all of that into the agent made its behaviour depend on whatever's installed globally that day — non-reproducible, and full of tools that fail or mislead. Now only a **curated allowlist** is bound (each read-only), default just `.credentials.json` — verified to be the only file `claude` needs to authenticate (it auths with that file alone even when the config-dir root is non-writable). Widen deliberately via `DEVCLAW_SANDBOX_CLAUDE_ALLOWLIST` (comma-separated entries under `~/.claude`). The writable scratch tmpfs overlays (`session-env/`, `shell-snapshots/`) are preserved. `docker run` argv assembly is factored into pure `_build_docker_args` / `_build_claude_mounts` so the mount posture is unit-tested without docker (`tests/test_sandbox_isolation.py`).
- **Abort handling — a kill switch for tasks and programs (16 tools total).** Two new MCP tools: `cancel_task(task_id)` and `cancel_program(program_id)`. Until now DevClaw was crash-safe (`recover()` revives orphaned `running` tasks) but had **no way to deliberately stop** in-flight work — no tool, no terminal state, and crash recovery would have resurrected anything you killed. New terminal status **`cancelled`** (distinct from `failed`) for both tasks and programs: it's terminal, so `recover()` (which only reaps `running` rows) leaves an abort aborted across restarts. Cancellation actually tears down the live run — `TaskQueue` now indexes each running task's `asyncio.Task` so cancel can reach in and cancel it, and `run_sandcastle` grew a `finally` that `docker rm -f`s the named container (killing the `docker run` client alone doesn't stop the container). Cancelling a program stops scheduling, cancels pending children, and tears down running ones; cancelling a single child sticky-cancels its program (a hole in the DAG blocks its dependents — same shape as sticky-failure). Covered by `tests/test_cancel.py`.
- **Per-role model tiering.** Cognition no longer runs everything on the account-default model (Opus) — each role binds its own tier via env, so an autonomous run stops burning the Pro/Max quota where a lighter model suffices (no API key → the limit is the session quota, not a bill; and lighter models run faster). New env knobs: `DEVCLAW_PLANNER_MODEL` (default `opus` — planning is rare + high-leverage), `DEVCLAW_GRILL_MODEL` (`sonnet`), `DEVCLAW_JUDGE_MODEL` (`haiku`), and `DEVCLAW_EXEC_MODEL` (`claude-sonnet-4-6` — the OpenHands coding agent, which is the token/quota bulk; set `claude-opus-4-8` to opt a run up to Opus). Two chokepoints: host calls thread `--model` through `planner.call_claude` (new `model` param + `claude_with_model` factory binding each role's default caller); the sandbox path passes the model in the runner payload → `ACPAgent(acp_model=…)` (Claude ACP selects it via session `_meta`). Empty value → account default (no regression). Covered by `tests/test_model_tiering.py`. Verified live: host roles resolve to distinct models (`opus`→claude-opus-4-8, `sonnet`→claude-sonnet-4-6, `haiku`→claude-haiku-4-5 via `modelUsage`), and the in-sandbox `acp_model` is honored — the identical trivial task costs ~5.9× more pinned to Opus than to Sonnet, matching the model price ratio.
- **Host engine (`DEVCLAW_ENGINE=host`).** Runs OpenHands directly on the host (no docker), using the host's `claude` + `~/.claude` — for environments without docker (dev/CI, live validation). ⚠ **No sandbox isolation** (the agent runs as your user with full filesystem access); production uses the docker sandbox. Factored the shared runner-protocol reader into `devclaw/runner_io.py` so both engines use it. Three engine modes now: default (sandbox), `host`, `stub`.
- **Steer inbox — redirect a running build (build-from-scratch step 5).** New `steer(project_id, message)` MCP tool (14 tools total) injects mid-build direction (e.g. "use Postgres, not SQLite") into the project's **not-yet-started** tasks — running/done tasks are untouched, so a long build responds to direction without restarting. Recorded in the project's `steer_log` + an append-only `steer-inbox.jsonl`. Task mutation goes through `TaskQueue.steer_program` (single-writer); `StateStore.append_note_to_pending_tasks` does the pending-only update.

### Fixed
- Eval harness bugs surfaced by the stub validation: `evals/run.py` passed a **relative** `accept.sh` path to a subprocess whose cwd is the workspace (→ phantom acceptance failures), and `accept.sh` assumed a bare `python` on PATH. Now the accept path is absolute and the check runs under the harness's own interpreter (`$EVAL_PYTHON`); the golden project's contract is `python3 -m jyq`.

### Added
- **Stub-engine harness validation (`DEVCLAW_ENGINE=stub`).** A deterministic engine + cognition (`devclaw/stub_engine.py`) the server wires in place of OpenHands + claude, so the *entire* eval pipeline (`build_project → grill → approve → build → score → archive`) runs end-to-end with no docker and no claude — proving the plumbing so a live failure isolates to agent quality. For the golden project the stub writes a genuinely-working `jyq`, so the green path (acceptance pass) is provable offline (`acceptance_pass_rate = 1.0`).
- **Failure-analysis judge for the eval harness.** `evals/run.py --judge` runs a second `claude` pass after each build that reads the spec + task DAG + event digest + acceptance result and buckets the run into a fixed failure-mode vocabulary (`success`, `planning_error`, `incomplete_build`, `constraint_violation`, `acceptance_gap`, `engine_failure`, `stuck`, `other`) with a diagnosis + a concrete suggestion. `summarize_verdicts` rolls N runs into a `by_category` breakdown + `top_failure_mode`, so failures aggregate and point at where to spend the next polish pass — automating the "analyze what went wrong" loop. New `devclaw/eval_judge.py` (unit-tested; cognition stubbed).
- **Eval harness — golden-project build smoke test.** `evals/run.py` drives the real pipeline (`build_project → grill → approve → build`) N times against a fixed project, with the grill answered from a script (`answers.txt`) so the spec is held constant and the *build* is what's measured. Each run is graded against a hard acceptance check (`accept.sh`) and archived; `devclaw/evals.py` (unit-tested) scores per-run (acceptance, milestone-completion %, task counts, wall-time, stuck) and aggregates N runs into a **pass-rate** — the success metric, comparable across git SHAs to measure progress. Ships one golden project (`evals/json-yaml-cli`, a JSON↔YAML round-trip CLI). See `evals/README.md`. This is the project's measurable answer to "does OpenHands have the skill-quality for autonomous work?"
- **Elicitation grill — build a project from scratch (step 4).** Four new MCP tools: `build_project(idea, workspace_dir)` starts a project and DevClaw **grills you one question at a time** (each with a recommended answer) to a shared spec; `answer_question(project_id, answer)` walks the interview; `get_project(project_id)` shows state across phases; `approve_spec(project_id)` plans the spec into a milestone DAG and starts the build. The grill methodology adapts [Matt Pocock's MIT `grill-me`](https://github.com/mattpocock/skills); cognition runs via `claude` (mechanism/cognition split), capped at `DEVCLAW_MAX_GRILL_QUESTIONS` (default 20). Project idea/transcript/spec persist to `$DEVCLAW_STATE/projects/<id>/` (human-readable). New `devclaw/{elicitation,project_store,project_service}.py`; approval hands an already-planned DAG to the executor via `TaskQueue.start_planned_program` (no plan-time recovery edge case).
- **Plan-from-spec (build-from-scratch step 3).** New `planner.plan_spec(spec, workspace_dir)` decomposes an *approved spec* (the shared understanding from the elicitation grill) into a **milestone-ordered DAG** — grounded in the spec's milestones, acceptance criteria, scope, and constraints, distinct from the bare-goal `plan_goal`. Tasks now carry an optional `milestone` (new SQLite column + forward-compat ALTER, surfaced in `get_program`/`list_tasks`), persisted by the queue. The `build_project` tool that produces the spec lands in step 4.
- **Durability + crash recovery (build-from-scratch step 2).** Scheduling is now reconciled from DB state by a single `_pump`, so concurrency is crash-consistent (derived from `running` rows, not an in-memory counter). On startup `recover()` resets tasks orphaned in `running` by a dead process back to `pending` (logged as a `reaped` event); a **heartbeat tick** (`DEVCLAW_TICK_SECONDS`, default 10s) advances DAGs and resumes recovered work from disk, so a multi-day build survives restarts. A **cheap-idle guard** makes an idle tick ~free (one COUNT). New **global concurrency cap** `DEVCLAW_MAX_CONCURRENT` (default 4) with backpressure, alongside the per-program cap.
