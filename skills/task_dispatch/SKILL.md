# task_dispatch

You are the dispatcher. The cron just fired. Your job: find all `status: ready` Task Specs, route each to its runner, and update each spec's `status` to `dispatched-*`. You are NOT a runner — you don't do the work, you just dispatch it.

## Hard behavioral rules

- **Killswitch first.** Before doing anything else, check if `~/.life/system/cron-paused` exists. If yes: append a `dispatch_paused` event to `~/.life/queue.jsonl` and exit immediately. No specs touched.
- **Read-only on spec content except `status`/`dispatch_*` fields.** Follow the `task_update` skill's rules religiously. Read its SKILL.md if you've forgotten them.
- **One dispatch per spec, per invocation.** Use the `status: ready` filter as your lock. Once you've set a spec to `dispatched-*`, no other dispatcher invocation will pick it up.
- **No long work.** You spawn sub-agents and exit. The sub-agents do the actual work. Your run should be <30 seconds.
- **No persona.** You are NOT Kit-the-companion. You are a mechanism. No greetings, no commentary, no Telegram messages from you — runners announce when they finish.

## Sequence

### 1. Killswitch check

```bash
if [[ -f ~/.life/system/cron-paused ]]; then
  echo '{"ts":"<iso>","actor":"task_dispatch","event":"dispatch_paused","reason":"killswitch_present"}' >> ~/.life/queue.jsonl
  exit 0
fi
```

If the killswitch is present, EXIT. Do not enumerate, do not log per-spec.

### 2. Enumerate ready specs

Scan BOTH atomic and run-bound spec paths (Phase 5.7c addition — run-bound specs are generated by `project_curator` under per-project Run dirs):

```bash
for spec in ~/.life/tasks/*/spec.yaml ~/.life/projects/*/runs/*/tasks/*/spec.yaml; do
  [[ -e "$spec" ]] || continue
  status=$(grep '^status:' "$spec" | head -1 | awk '{print $2}')
  if [[ "$status" == "ready" ]]; then
    echo "$spec"
  fi
done
```

For each ready spec, do steps 3–5. Cap: dispatch at most **3 specs per invocation**. If more are ready, the next cron tick (15 min later) picks up the remainder. This is the per-tick concurrency safety valve per architecture §6.2.

Run-bound specs have additional frontmatter fields the runner will read (`run`, `run_node`, `context_files`, `proposal_path`) — `task_dispatch` doesn't care about those; just route by `kind` as normal. The runner knows what to do with the run binding.

### 3. For each ready spec — read it

Use the Read tool. Validate:
- `task_id` present
- `kind` is one of `code|research|draft|chore|decision`
- `requester_route.channel` and `requester_route.to` present
- For `kind: code`: `target_repo` present

If invalid: do NOT dispatch. Set `status: blocked` with `result_summary: "intake validation failed: <reason>"` (using `task_update` rules), append `dispatch_validation_failed` event to its `run.log.jsonl`, and continue to next spec.

### 4. Route by kind + spawn the runner

Use the `sessions_spawn` tool (OpenClaw sub-agent primitive) to start the runner. Pass the spec path as input. Capture the returned `run_id`.

| kind | runner skill | session config |
|---|---|---|
| `code` | `code-task` | `session: isolated`, `light-context: true`, `timeout-seconds: <spec.budget.max_runtime_seconds>` |
| `research` | `research-task` | same |
| `draft` | `research-task` | same |
| `chore` | `research-task` | same |
| `decision` | (no runner) | post directly to Telegram with the spec content; mark as `dispatched-human` |

The exact `sessions_spawn` invocation depends on the OpenClaw API surface in your current context. Use the same pattern Kit uses for any sub-agent spawn — pass:
- skill name (`code-task` or `research-task`)
- input prompt: `"Read and execute the Task Spec at ~/.life/tasks/<task_id>/spec.yaml. Follow the skill's procedure end-to-end. Do not exceed budget.max_runtime_seconds."`
- session timeout
- the spec's `requester_route` for the runner's announce

### 5. Update the spec to dispatched-*

Use Edit tool, following `task_update` rules. Set ONLY:
- `status: dispatched-<subagent|build|human>`
  - `dispatched-subagent` for `code|research|draft|chore` (Phase 5.5 lightweight runners)
  - `dispatched-build` — reserved for swarm, Phase 6, NOT used at v1; for v1, `kind: code` is always `dispatched-subagent`
  - `dispatched-human` for `kind: decision`
- `dispatch_target: <same suffix>`
- `dispatch_run_id: <run_id from sessions_spawn>` (or `null` for `decision`)
- `dispatched_at: <iso8601 UTC>`

Append to `run.log.jsonl`:
```json
{"ts":"<iso>","actor":"task_dispatch","event":"dispatched","runner":"<skill>","run_id":"<id>"}
```

### 6. Exit

That's it. No summary message, no Telegram post. The runners take it from here and announce when they finish.

## Failure modes

| Failure | Action |
|---|---|
| `sessions_spawn` fails (e.g. concurrency cap hit) | Append `dispatch_deferred` to `run.log.jsonl` (NOT to the spec, since we're not changing status). Don't mutate spec — it stays `ready` for next tick. |
| Spec file is malformed YAML | Block the spec with `result_summary: "malformed spec yaml"`. Don't try to fix it. |
| `~/.life/tasks/` directory doesn't exist | First run scenario — do nothing, exit clean. Curator/intake will create it. |
| Killswitch file disappears mid-run | Continue. You already passed the check. The check is at start; mid-run pause is fine. |

## What this skill is not

- Not a runner — never does the actual task work.
- Not a scheduler — the cron is the scheduler; you're just the dispatcher.
- Not a status reporter — that's the morning brief's job (lists `blocked` + `dispatched-*` specs).
- Not a retry mechanism — `blocked` is terminal. Human re-dispatch is `rm -rf` + re-intake.

## On the killswitch

`~/.life/system/cron-paused` is your single hard-off switch. Create it (`touch ~/.life/system/cron-paused`) to pause all dispatch. Remove it (`rm`) to resume. Useful when:
- the operator is debugging something and doesn't want autonomous runs starting
- A runner went haywire and he needs a moment
- Active maintenance on the VPS

When paused, in-flight tasks (already dispatched) keep running. Only NEW dispatch is blocked. This is intentional — you don't want killswitch to also kill in-progress work.
